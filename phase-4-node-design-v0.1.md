# 阶段 4 爬塔节点设计 v0.1

版本：0.1
状态：阶段 4A/4B/4C 已实现并通过自动化；数值待 Unity 人工验收调整
关联文档：`game-design.md`、`development-plan.md`、`shop-economy-design-v0.1.md`、`phase-3-build-design-v0.1.md`、`phase-4a-technical-design-v0.1.md`、`phase-4b-technical-design-v0.1.md`

## 1. 阶段目标

阶段 4 的目标不是继续扩充卡牌数量，而是把现有商店、构筑和战斗系统串成一局可以通关或失败的最小爬塔流程。

本阶段需要验证：

- 玩家是否会为了风险、恢复、经济和高价值奖励选择不同路线。
- 商店构筑、战斗压力和节点奖励能否形成连续节奏。
- 战斗失败但未死亡时，玩家是否能理解损失并继续调整阵容。
- 精英和 Boss 奖励是否足以影响后续构筑，而不是只有数值增长。
- 一局三层流程能否控制在 10-20 分钟内。

阶段 4 完成后，玩家应能从空阵容开始，经过三层地图，最终击败第三层 Boss 通关，或因生命归零失败。

## 2. 设计状态说明

本文中的规则分为两类：

- 已继承规则：来自现有策划和阶段 1-3 已确认规则，阶段 4 默认继续使用。
- 阶段 4 基线：本文件已经确认的地图、节点、奖励、生命和流程规则，直接作为第一版实现依据。

本文优先保证流程能闭环、状态不会卡死、规则容易解释。随机地图、遗物、复杂 Boss 专属机制和正式数值平衡不属于第一批节点实现。

## 3. 已确认基线总览

| 项目 | 已确认基线 |
| --- | --- |
| 地图结构 | 三层固定有向图，每层 6-8 个可见节点，单次实际经过 3-4 个节点（含 Boss） |
| 地图信息 | 进入本层时显示完整节点类型和连线，不隐藏后续路线 |
| Boss | 每层固定一个 Boss，第三层 Boss 胜利后通关 |
| 初始生命 | 20 点，作为配置项保留调整空间 |
| 战斗扣血 | 敌方存活随从数 + 敌方最高等级 + 节点伤害修正，最低 1 点 |
| 平局 | 区分双方同时倒下与回合上限两种原因；第一版均扣除 1 点生命并按未胜利处理 |
| 流程回合 | 每选择一个节点或重试一次 Boss，`RunTurn` 推进一次；是否打开商店由节点类型决定 |
| 小怪失败 | 扣血后节点完成，无奖励，生命未归零则继续前进 |
| 精英失败 | 额外扣血，节点完成，无精英奖励 |
| Boss 失败 | 额外扣血，Boss 节点不完成；生命未归零则进入下一轮商店并重试 |
| 强化节点 | 使用“锻造台”，只改造已有随从，不卖卡、不使用金币；每层最多一个 |
| 恢复点 | 在“恢复 6 点生命”和“最大生命 +2 并恢复 2”之间二选一，不承担随从强化 |
| 卡牌奖励 | 使用先进先出的独立奖励队列，在后续商店领取或跳过，避免备战区已满导致流程卡死 |
| 地图生成 | 第一版使用固定地图和固定 Boss，通过统一地图提供接口为随机生成预留扩展口 |

## 4. 地图和节点通用规则

### 4.1 地图结构

第一版每层地图使用固定的有向无环图。节点只允许从左向右或从上向下推进，不能回退到已完成节点。

第一版每层结构：

1. 一个层起点。
2. 第一层起点只连接一个强制小怪节点；第二、三层可从起点直接提供两个可选节点。
3. 强制首战或层起点之后提供两个分支，后续可继续提供两个或三个分支节点。
4. 路线在 Boss 前合流。
5. 一个固定 Boss 节点作为本层终点。

“每层 6-8 个节点”指地图上的节点总数，不代表玩家必须访问全部节点。单条路线实际访问 3-4 个节点并包含 Boss，完整三层单局约访问 10-12 个节点，避免每个战斗节点都包含完整商店时导致单局时间失控。

第一层开局采用以下固定边界：

- 玩家以空阵容、3 金币和 20/20 生命开始。
- 第一层必须先进入一个不可绕过的普通战斗节点，该节点在战斗前打开本局第一个商店。
- 首个遭遇只配置一个低强度 1 级敌方随从，用于验证玩家完成了最基础的购买和上场操作。
- 只有首战结算完成后才解锁第一次路线分支。
- 第二、三层已经有成型阵容，可以在进入楼层后立即提供路线分支。

### 4.2 地图生成扩展口

固定地图只是一种地图数据来源，流程层和 UI 不应直接依赖固定 JSON 文件或具体生成算法。

地图统一通过以下接口提供：

```csharp
public interface IMapProvider
{
    MapDefinition CreateMap(MapRequest request);
}
```

第一版实现：

- `FixedMapProvider`：按层数读取已经配置和校验过的固定地图。
- 固定地图、节点状态推进、场景流程和 UI 通过 `MapDefinition` 工作，不感知数据来自固定配置还是生成器。

后续可以增加：

- `RandomMapProvider`：根据单局种子、层数、节点权重和路线约束生成地图。
- `MapRequest` 至少包含单局种子、层数、难度和允许节点类型。
- 随机生成结果必须通过与固定地图相同的图结构校验，保证所有路线可到达唯一 Boss。

`RunSession` 只依赖 `IMapProvider`，并保存生成结果和随机种子。切换随机地图时不修改节点结算、地图 UI、存档结构和流程状态机。

### 4.3 节点状态

节点统一具有以下状态：

- `Locked`：当前不可选择。
- `Reachable`：从当前节点可以进入。
- `Current`：玩家正在处理该节点。
- `Resolved`：节点已经完成，不能再次进入。

选择一个 `Reachable` 节点后：

- 该节点变为 `Current`。
- 同一分支层级中未选择的互斥节点继续保留在地图上，但不能回头进入。
- 只有当前节点完整结算后，后继节点才变为 `Reachable`。
- 奖励选择、事件选择或战斗结算未完成时，不能选择下一个节点。

### 4.4 节点通用生命周期

所有节点遵循统一生命周期：

1. 从地图选择节点。
2. 锁定地图操作并进入节点。
3. 根据节点类型执行商店、战斗、事件、强化或恢复流程。
4. 提交节点结果。
5. 结算生命、奖励和延迟资源。
6. 将节点标记为 `Resolved`，或按 Boss 失败规则保留为 `Current`。
7. 解锁后继节点或进入下一层。

节点处理必须是可恢复的显式状态，不能仅依赖当前显示的 Unity 场景判断流程进度。

每次节点进入或 Boss 重试都创建唯一的 `NodeAttemptId`。同一尝试至少记录以下提交标记：

- `runTurnCommitted`
- `battleSettled`
- `healthDamageApplied`
- `rewardGenerated`
- `nodeResolved`

场景重载只能恢复同一个尝试，不能创建新尝试。Boss 失败后的再次挑战创建新的 `NodeAttemptId`。`RunTurn`、战斗结算、生命扣除、奖励生成和节点完成分别按对应标记保证每个尝试最多提交一次。

### 4.5 节点、流程回合与商店

已确认商店阶段同时承担阵容准备功能。阶段 4 将地图流程回合与商店开启次数分开：

- `RunTurn` 初始为 0；选择并成功进入地图节点时，在创建节点尝试的同一个原子操作中推进一次。
- Boss 首次进入沿用节点进入产生的尝试；Boss 失败后的每次重试创建新尝试，并各推进一次 `RunTurn`。
- 进入小怪、精英或 Boss 节点时，先开始一轮正常商店阶段。
- 玩家结束商店后锁定阵容，立即进入该节点的战斗。
- 事件、强化和恢复节点不开始商店阶段，但已经推进的 `RunTurn` 仍会影响下一次商店预算，并视为跳过了一次自然升级降价机会。
- 商店预算固定为 `min(RunTurn + 2, 10)`，不按累计打开商店的次数计算；因此第 1、2、3 个节点尝试的预算分别为 3、4、5 金币。
- 战斗节点在商店正常结束且本回合未升级时，自然升级降价推进一次；非战斗节点在进入时推进一次等价的自然降价。两者都必须按 `RunTurn` 幂等处理。
- 事件或奖励提供的额外升级折扣可以叠加，并持续保留到下一次成功升级；成功升级后，自然降价和所有额外折扣一起清零。
- 最终升级费用最低为 1 金币。
- 冻结状态可以跨地图节点保留到下一次商店阶段。
- “下个商店阶段”类奖励必须在预算重置和商店补货完成后应用。

现有 `ShopSession.Round` 不能继续同时代表商店开启次数和地图流程时间。阶段 4 实现时应由流程层传入当前 `RunTurn` 或对应预算上下文，避免事件、强化和恢复路线受到双重经济惩罚。

## 5. 单局公共状态

阶段 4 需要在现有 `RunSession` 基础上维护以下单局状态：

- 单局随机种子。
- 当前 `RunTurn`。
- 当前流程阶段。
- 当前层数。
- 当前节点 ID。
- 已完成节点集合。
- 当前可达节点集合。
- 当前生命和最大生命。
- 持续存在的 `ShopSession`。
- 当前遭遇配置。
- 当前战斗快照和战斗结果。
- 当前 `NodeAttemptId` 和节点尝试提交标记。
- 待选择奖励。
- 待领取卡牌奖励。
- 下个商店阶段金币奖励。
- 下个商店阶段免费刷新次数。
- 下次酒馆升级折扣。
- 小怪、精英、Boss 胜负统计。
- 三连次数和单局用时。

第一版只要求内存态，不要求退出游戏后恢复单局。正式存档仍属于后续阶段。

## 6. 玩家生命和战斗伤害

### 6.1 生命规则

- 初始生命和最大生命均为 20。
- 恢复不能超过最大生命。
- 战斗伤害可以使生命降至 0。
- 生命降至 0 时立即进入失败结算，不能继续领取奖励或选择节点。
- 主动事件的生命代价不能使玩家生命降至 0；支付后至少保留 1 点生命。

### 6.2 扣血公式

基础公式：

`战斗伤害 = max(1, 敌方存活随从数量 + 敌方存活随从最高等级 + 节点伤害修正)`

节点伤害修正初值：

| 节点类型 | 修正 |
| --- | ---: |
| 小怪 | 0 |
| 精英 | +2 |
| Boss | +4 |

补充规则：

- 金色随从仍使用原配置等级计算，不额外翻倍。
- Token 计入存活随从数量，等级按其配置等级计算。
- 敌方无存活随从时不会产生失败伤害。
- 平局固定扣除 1 点生命，不使用存活随从公式。
- 战斗结果必须保留 `BattleOutcomeReason`：`Victory` 表示一方获胜并由 `Winner` 指明胜者，`MutualElimination` 表示双方同时倒下，`RoundLimit` 表示达到回合上限。
- `MutualElimination` 和 `RoundLimit` 第一版都扣除 1 点生命，但必须保留不同原因，便于后续分别调参和展示。
- 战斗伤害只修改玩家单局生命，不修改持有随从。

该公式的优点是伤害来源可以在结算界面直接拆解显示。具体数值需要在阶段 4 完成后根据一局实际战斗记录调整。

## 7. 小怪节点

### 7.1 设计目的

- 提供主要的基础战斗节奏。
- 检验玩家当前阵容是否达到本层基本战力。
- 提供低强度构筑资源，但不直接奖励金币。
- 让玩家理解失败扣血后仍可继续调整阵容。

### 7.2 进入流程

1. 玩家在地图选择小怪节点。
2. 加载节点绑定的普通遭遇配置。
3. 开始一轮正常商店阶段。
4. 玩家结束商店，生成玩家战斗快照。
5. 将遭遇配置转换为敌方战斗快照。
6. 进入自动战斗。

### 7.3 胜负结算

胜利：

- 不直接获得金币。
- 生成一项轻量普通奖励。
- 处理或确认奖励后，小怪节点完成。

失败：

- 按小怪公式扣除生命。
- 不生成普通奖励。
- 生命大于 0 时小怪节点仍然完成，可以继续前进。

平局：

- 扣除 1 点生命。
- 不生成奖励。
- 节点完成。

### 7.4 第一版普通奖励池

每次胜利随机生成一项，不再打开三选一界面，以控制节点时长：

- 下个商店阶段获得 1 次免费刷新。
- 下个商店阶段额外获得 1 金币。
- 获得一张不高于当前酒馆等级的普通商店法术，进入奖励暂存区。

地图和战斗结果界面需要明确显示本次获得的奖励。

### 7.5 敌人设计要求

- 每个小怪遭遇使用固定阵容，第一版不随机组牌。
- 敌方阵容不消耗玩家有限随从牌池。
- 同一层至少准备 2 套普通遭遇，避免每条路线完全相同。
- 第一层使用基础身材、嘲讽和护盾。
- 第二层开始加入溅射和简单亡语。
- 第三层要求玩家关注站位和核心随从保护。

### 7.6 验收标准

- 小怪节点能够完整执行商店、战斗、结算和返回地图。
- 胜利只发放一次奖励。
- 失败只扣除一次生命。
- 失败但未死亡时可以继续选择后继节点。
- 场景重载不会重复开始商店或重复结算战斗。

## 8. 精英节点

### 8.1 设计目的

- 为路线选择提供明确的高风险高收益选项。
- 用具有主题的敌方阵容检查玩家阵容弱点。
- 提供足以改变构筑方向的三选一奖励。

### 8.2 进入流程

精英节点与小怪节点使用相同的“商店准备后战斗”流程，但必须在地图上使用独立图标，并在进入前显示：

- 精英名称。
- 主题关键词。
- 风险提示。
- 奖励类型提示。

第一版不要求预览完整敌方阵容和精确数值，避免路线选择变成计算题。

### 8.3 胜负结算

胜利：

- 不直接奖励金币。
- 生成三个互不重复的奖励候选。
- 玩家必须选择一个或主动跳过，之后节点完成。

失败：

- 使用基础扣血公式并增加精英伤害修正。
- 不生成精英奖励。
- 生命大于 0 时节点完成，可以继续前进。

平局：

- 按精英失败处理，但只扣除 1 点基础平局伤害，不增加精英修正。
- 不生成奖励。

### 8.4 第一版精英奖励池

候选从以下类别中生成，三个候选尽量来自不同类别：

- 一张当前酒馆等级的普通随从。
- 一张不高于当前酒馆等级的普通商店法术。
- 下个商店阶段获得 2 次免费刷新。
- 下个商店阶段额外获得 3 金币。
- 指定一个战斗区随从永久获得 +1/+2。

涉及卡牌的奖励选择后进入奖励暂存区，不要求备战区当场有空位。

### 8.5 第一批精英主题

第一版每层配置一个精英主题：

1. 嘲讽壁垒：多个嘲讽和护盾单位，检验持续输出。
2. 裂阵猎手：使用溅射单位，检验站位和核心保护。
3. 亡语群落：死亡后召唤 Token，检验残局能力。

第一版只使用现有战斗关键词和效果，不增加精英专属战斗规则。

### 8.6 验收标准

- 精英节点在地图上与小怪节点清晰区分。
- 精英失败伤害高于同层小怪。
- 精英胜利进入阻塞式三选一奖励流程。
- 选择、跳过或空间不足都不会卡死流程。
- 未选择前不能进入下一个节点。

## 9. 强化节点：锻造台

独立商店节点从阶段 4 第一版移除。所有购买、刷新、升级、冻结、法术和阵容准备继续只发生在战斗节点前的商店阶段。

强化节点定名为“锻造台”。它提供与商店有明确区分的阵容塑形能力，并作为阶段 4 第一版正式节点类型。

### 9.1 与其他节点的功能边界

强化节点与商店的区别：

- 不生成随从或法术商品。
- 不使用金币。
- 不允许购买、出售、刷新、升级酒馆或冻结。
- 不推进酒馆商品池，也不消耗商店法术池。
- 只修改玩家已经持有并打入战斗区的随从。

强化节点与事件的区别：

- 强化收益确定，不包含生命代价、随机负面结果或经济交换。
- 玩家明确选择强化方式和目标，不依赖隐藏事件结果。

强化节点与恢复点的区别：

- 强化节点只处理阵容战力和关键词。
- 恢复点只处理玩家当前生命和最大生命。

### 9.2 节点流程

1. 进入强化节点，推进 `RunTurn`，但不打开商店。
2. 展示该固定节点配置的 3 个强化配方。
3. 玩家选择一个配方。
4. 玩家选择一个最终战斗区随从作为目标。
5. 统一验证配方、目标和实际收益。
6. 原子提交永久强化。
7. 播放强化反馈并完成节点。

玩家可以主动离开并放弃强化，避免没有合法目标时流程卡死。

### 9.3 第一批强化配方

第一版固定使用现有运行时已经支持的以下属性和关键词：

| 配方 | 效果 | 出现层数 |
| --- | --- | --- |
| 均衡锻造 | 指定随从永久获得 +2/+2 | 1-3 |
| 重甲铭刻 | 指定随从永久获得 `Shield` | 1-3 |
| 守卫铭刻 | 指定随从永久获得 `Taunt` | 1-3 |

限制：

- 目标必须位于最终战斗区，不能选择备战区。
- 已有相同永久关键词时，对应铭刻配方视为零收益并禁止选择。
- 不允许授予 `Battlecry` 或 `Deathrattle`，因为关键词本身没有绑定效果时没有意义。
- 同一个随从可以在不同锻造台重复强化，第一版不增加“已锻造”状态。
- 同一个永久关键词不能重复授予，但“均衡锻造”的 +2/+2 可以重复叠加。
- 金色随从继承合成素材已有的永久属性和关键词，合成后仍可继续接受强化；永久属性不会因金色状态再次翻倍。
- 每个强化节点只提交一次配方。
- 每层最多配置一个强化节点，避免免费永久强化压过商店和三连价值。
- `Cleave` 铭刻暂缓，不进入阶段 4 第一批配方。

### 9.4 候选配置与后续扩展

第一版固定地图为每个强化节点配置固定配方列表，不在进入节点时随机抽取，保证平衡结果可复现。

后续随机地图可以根据以下信息抽取配方：

- 当前层数。
- 玩家战斗区已有关键词。
- 本局已经访问的强化节点数量。
- 配方稀有度和互斥标签。

候选生成只决定展示什么，不改变统一的目标验证和结算入口。

### 9.5 风险

- 免费关键词可能比精英奖励更强，导致强化路线成为固定选择。
- `Cleave` 等关键词可能使部分随从的定位发生过大变化。
- 单纯数值强化容易退化成“免费法术”，与事件奖励缺乏差异。
- 如果强化配方过多，会提前扩张完整效果 DSL 的范围。

第一版只实现“均衡锻造 + Shield + Taunt”。`Cleave` 作为第二批候选，在实际战斗验证后决定是否开放。

### 9.6 验收标准

- 强化节点不创建商店、不修改金币和酒馆状态。
- 每个节点最多成功强化一次。
- 非法目标、重复关键词和零收益配方不会部分生效。
- 强化永久保留到后续商店和战斗。
- 重复访问不同锻造台时，同一随从可再次获得数值强化，但不能重复获得已有关键词。
- 没有合法目标时可以离开节点。
- 场景重载不会重复应用强化。

### 9.7 配置

强化配方不写死在地图节点或 UI 中。`EnhancementRecipeConfig` 至少包含：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | string | 配方唯一 ID |
| name | string | 显示名称 |
| action | string | `ModifyStats` 或 `GrantKeyword` |
| attack | int | 永久攻击加成 |
| health | int | 永久生命加成 |
| keyword | string | 要授予的永久关键词 |
| minimumFloor | int | 最低出现层数 |
| weight | int | 后续随机候选权重 |
| enabled | bool | 是否进入候选池 |

`EnhanceNodeConfig` 包含：

- 节点内容 ID。
- 第一版固定展示的配方 ID 列表。
- 候选数量。
- 是否允许跳过。
- 后续随机生成使用的配方标签或权重组。

固定地图直接配置配方 ID；后续随机地图可以只配置候选规则。两种方式最终都生成同一种强化选择状态和 UI 数据。

## 10. 事件节点

### 10.1 设计目的

- 提供不依赖战斗的风险收益选择。
- 改变下一轮商店或当前阵容状态。
- 调整连续战斗带来的节奏疲劳。

### 10.2 通用规则

- 事件节点不开始商店阶段，但进入节点时正常推进 `RunTurn`。
- 地图上只显示通用事件图标，进入后才揭示具体事件。
- 每个事件提供 2-3 个明确选项。
- 选项在提交前显示确定的代价和收益；第一版避免隐藏随机惩罚。
- 事件选择作为原子操作，失败时不能只支付代价而不获得收益。
- 事件只能结算一次。
- 事件未选择前不能返回地图或进入其他节点。
- 如果所有收益选项都不合法，必须提供“离开，不发生任何事”。

### 10.3 第一批五个事件

#### 事件一：鲜血契约

- 选项 A：失去 3 点生命，下个商店阶段额外获得 4 金币。
- 选项 B：离开，不发生任何事。
- 当前生命小于或等于 3 时禁用选项 A。

#### 事件二：旧酒馆账本

- 选项 A：下次酒馆升级额外降低 2 金币。
- 选项 B：下个商店阶段获得 1 次免费刷新。
- 升级折扣和免费刷新都只应用一次。

#### 事件三：遗失的补给车

- 选项 A：下个商店阶段获得 2 次免费刷新。
- 选项 B：获得一张不高于当前酒馆等级的随机普通法术，进入奖励暂存区。

#### 事件四：星泉

- 选项 A：恢复 4 点生命。
- 选项 B：下个商店阶段额外获得 2 金币。
- 生命已满时禁用选项 A，并显示“生命已满”。

#### 事件五：危险招募

- 选项 A：失去 2 点生命，获得一个当前酒馆等级随从的三选一奖励。
- 选项 B：离开，不发生任何事。
- 当前生命小于或等于 2 时禁用选项 A。

### 10.4 验收标准

- 五个事件都能从配置加载并完整结算。
- 非法选项明确置灰并显示原因。
- 生命代价不会使玩家因主动选择直接死亡。
- 延迟资源只在下个商店阶段应用一次。
- 事件产生的卡牌奖励不会因备战区已满丢失。

## 11. 恢复点

### 11.1 设计目的

- 为高风险路线提供恢复机会。
- 让满生命玩家仍然能获得有价值的选择。
- 在立即恢复和提高后续生命容错之间制造取舍。
- 与强化节点分工：恢复点只处理玩家生命，不修改随从。

### 11.2 节点选项

进入恢复点后从以下选项中选择一个：

- 休息：恢复 6 点生命，不超过最大生命。
- 深度休整：最大生命永久增加 2，并恢复 2 点生命。
- 离开：不发生任何事。

恢复点不开始商店阶段，但进入节点时正常推进 `RunTurn`。

### 11.3 目标和收益规则

- 休息在生命已满时禁用，不允许提交零收益恢复。
- 深度休整先增加最大生命，再恢复生命，因此满生命时仍然具有实际收益。
- 最大生命增加属于本局永久状态，只持续到本局结束。
- 选择提交后恢复点立即完成，不能反复切换选项。

### 11.4 验收标准

- 恢复不会超过最大生命。
- 最大生命增加会保留到本局结束。
- 满生命时不能选择零收益休息，但仍可深度休整或离开。
- 恢复点只能结算一次。

## 12. Boss 节点

### 12.1 设计目的

- 检验本层构筑是否达到要求。
- 形成每层明确的阶段终点。
- 通过高价值奖励推动下一层构筑变化。

### 12.2 地图规则

- Boss 节点固定为每层终点。
- 进入本层时显示 Boss 名称、主题关键词和简短提示。
- 所有合法路线最终都必须到达 Boss。
- Boss 节点不能被跳过。

### 12.3 战斗流程

1. 进入 Boss 节点。
2. 开始一轮正常商店阶段。
3. 结束商店并锁定阵容。
4. 生成固定 Boss 遭遇快照。
5. 进行自动战斗。
6. 根据胜负推进楼层、重试或结束单局。

### 12.4 胜负结算

胜利：

- 第一、二层生成 Boss 三选一奖励。
- 奖励处理完成后进入下一层地图。
- 第三层 Boss 胜利后直接进入通关结算，不再发放会修改单局状态的功能奖励。

失败：

- 使用基础扣血公式并增加 Boss 伤害修正。
- 生命归零时单局失败。
- 生命大于 0 时 Boss 节点保持 `Current`，不能选择其他路线。
- 推进一次 `RunTurn`，重新开始正常商店阶段，玩家调整阵容后再次挑战同一 Boss。
- 每次重试都会增加商店预算进度，但同时承受新的 Boss 生命风险。

平局：

- 扣除 1 点生命。
- Boss 节点不完成，需要重试。

允许 Boss 重试会带来“故意失败换商店轮次”的风险。第一版不设置固定重试次数，先通过较高 Boss 伤害和有限生命自然限制，并在测试中记录是否形成可利用策略。

### 12.5 第一版 Boss 奖励池

Boss 三选一候选来自不同类别：

- 一个不高于“当前酒馆等级 + 1”的高阶随从。
- 一张高价值法术。
- 下次酒馆升级额外降低 3 金币。
- 下个商店阶段额外获得 5 金币。
- 指定一个战斗区随从永久获得 +2/+2。

阶段 4 不引入遗物和永久扩大战斗区/备战区上限，避免新增独立系统。

### 12.6 第一批 Boss 主题

1. 第一层 Boss：铸魂守门者。以嘲讽和护盾为主，检验基础持续输出和目标随机规则。
2. 第二层 Boss：裂阵兽王。以溅射和高攻击为主，检验站位与核心保护。
3. 第三层 Boss：荒灵母巢。以亡语召唤和残局铺场为主，检验完整阵容的持续作战能力。

Boss 第一版只组合现有随从和关键词，不实现多阶段血条、Boss 主动技能或战斗外规则。

### 12.7 验收标准

- Boss 胜利和失败不会重复结算。
- 第一、二层 Boss 胜利后正确进入下一层。
- 第三层 Boss 胜利后进入通关结算。
- Boss 失败但未死亡时只能重试 Boss，不能跳过。
- 重试会保留持有阵容、酒馆等级、牌池和永久强化。
- Boss 奖励在空间不足时可以暂存或跳过。

## 13. 奖励系统共用规则

### 13.1 奖励状态

阶段 4 区分两类奖励状态：

- `PendingRewardChoice`：正在等待玩家从候选中选择，属于阻塞流程。
- `PendingCardRewards`：已经选定但尚未进入备战区的卡牌奖励，可在下一商店领取。

规则：

- 奖励选择或跳过前不能选择地图节点。
- 资源和生命奖励在选择提交时立即或按“下个商店阶段”规则记录。
- 卡牌奖励选择后进入奖励暂存区，不直接要求备战区有空位。
- `PendingCardRewards` 使用先进先出队列，第一版不设置数量上限；玩家可以在领取前继续地图流程。
- 随从奖励从有限牌池抽取时，需要在候选生成时保留实体副本。
- 选中的随从奖励继续占用其保留的实体副本，直到领取、跳过或单局结束。
- 未选候选立即返还牌池；被跳过的已选随从奖励在跳过时返还牌池。
- 法术奖励不消耗有限随从牌池。
- 进入下一次商店后，必须按队列顺序领取或跳过全部待领取奖励，之后才允许结束商店。
- 单局通关、失败或主动终止时，必须释放所有尚未领取的随从奖励牌池预留。

### 13.2 延迟商店资源

地图阶段不能直接修改即将被下一轮预算重置覆盖的当前金币和免费刷新次数。`RunSession` 记录：

- `nextShopGoldBonus`
- `pendingFreeRefreshes`
- `nextUpgradeDiscount`

下次商店阶段的应用顺序：

1. 流程层按当前 `RunTurn` 开始商店，重置对应预算并生成商店。
2. 应用额外金币。
3. 应用免费刷新次数。
4. 应用升级折扣。
5. 打开奖励暂存区领取界面。
6. 允许正常商店操作。

每项延迟资源提交后记录已应用的 `RunTurn` 再清零；场景重载时先检查提交记录，不能只依赖“调用后清零”的时序防重。

### 13.3 奖励跳过

- 所有三选一奖励都提供“跳过”。
- 跳过不提供替代金币。
- 卡牌奖励暂存后也可以在商店主动跳过。
- 跳过有限牌池随从时返还其保留副本。
- 存在待领取卡牌奖励时，不能结束商店；领取或跳过队首后才处理下一项。

允许跳过的目的是消除备战区满、候选无收益或玩家不希望污染构筑时的流程死锁。

## 14. 遭遇配置方案

敌方阵容需要从 `BattleTestController` 的测试预设迁移为正式配置。

`EncounterConfig` 字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | string | 遭遇唯一 ID |
| name | string | 显示名称 |
| floor | int | 所属层数 |
| category | string | `Normal`、`Elite` 或 `Boss` |
| theme | string | 地图和预览使用的主题说明 |
| enemySlots | EnemySlotConfig[] | 最多 5 个敌方槽位 |
| damageBonus | int | 节点失败伤害修正 |
| rewardTableId | string | 胜利奖励表 ID |
| enabled | bool | 是否进入正式遭遇池 |

`EnemySlotConfig` 字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| slot | int | 0-4 槽位 |
| minionId | string | 随从配置 ID |
| golden | bool | 是否使用金色版本 |
| attackBonus | int | 遭遇专用攻击加成 |
| healthBonus | int | 遭遇专用生命加成 |
| permanentKeywords | string[] | 遭遇专用永久关键词，第一版可为空 |

第一版敌方阵容不消耗有限牌池，也不需要敌方备战区或商店状态。

## 15. 地图和节点配置方案

`MapNodeConfig` 字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | string | 节点唯一 ID |
| floor | int | 所属层数 |
| column | int | 地图布局列 |
| row | int | 地图布局行 |
| type | string | `Normal`、`Elite`、`Enhance`、`Event`、`Rest`、`Boss` |
| payloadId | string | 遭遇、事件或其他内容配置 ID |
| nextNodeIds | string[] | 可到达后继节点 |
| visible | bool | 是否在进入本层时显示 |
| enabled | bool | 是否启用 |

地图配置加载时必须校验：

- 节点 ID 不重复。
- 所有 `nextNodeIds` 都存在。
- 不存在环路。
- 每层至少有一个起始可达节点。
- 每层所有合法路线最终都能到达唯一 Boss。
- Boss 没有同层后继节点。
- `payloadId` 与节点类型匹配。

## 16. 流程状态

`RunPhase` 包含：

- `MapSelection`
- `EnteringNode`
- `Shop`
- `Battle`
- `BattleResult`
- `RewardChoice`
- `EventChoice`
- `EnhanceChoice`
- `RestChoice`
- `FloorComplete`
- `RunWon`
- `RunLost`

所有外部操作先验证当前 `RunPhase`。例如：

- 只有 `MapSelection` 可以选择节点。
- 只有 `Shop` 可以执行商店操作。
- 只有 `RewardChoice` 可以选择节点奖励。
- 只有 `EventChoice` 可以提交事件选项。
- 只有 `EnhanceChoice` 可以提交强化配方和目标。
- `RunWon` 和 `RunLost` 只能进入结算或重新开始。

场景只是状态的表现层。重新加载同一场景时，控制器必须从 `RunSession` 当前阶段重建界面，不能重新发放资源或重新结算节点。

## 17. UI 最小需求

### 17.1 地图界面

- 显示当前层数和玩家生命。
- 显示完整节点类型和连线。
- 区分锁定、可达、当前和已完成状态。
- 精英和 Boss 显示风险与主题提示。
- 当前存在阻塞奖励或选择时禁用地图点击。

### 17.2 战斗结果界面

- 显示胜利、失败或平局。
- 失败时拆解显示存活数、最高等级、节点修正和最终伤害。
- 显示扣血前后生命。
- 显示本次是否获得奖励。

### 17.3 奖励界面

- 显示候选的实际收益。
- 卡牌奖励显示名称、属性、等级和描述。
- 非法或暂时无法领取时显示原因。
- 提供选择和跳过入口。

### 17.4 单局结算界面

- 通关或失败结果。
- 到达层数和最终节点。
- 战斗胜负数。
- 精英和 Boss 击败数。
- 三连次数。
- 最终阵容。
- 单局用时。

第一版不要求正式地图动画、路线移动动画和奖励特效。

## 18. 自动化测试范围

### 18.1 EditMode

- 固定地图配置合法性。
- `FixedMapProvider` 和后续地图提供器返回相同结构契约。
- 节点可达状态推进。
- 互斥路线不能回退。
- 各节点只结算一次。
- 小怪、精英和 Boss 伤害公式。
- 生命归零进入失败状态。
- Boss 胜利推进楼层，失败保留当前节点。
- 延迟商店资源只应用一次。
- 奖励候选保留和返还牌池。
- 备战区满时卡牌奖励可以暂存和跳过。
- 事件生命代价和目标验证。
- 恢复和永久强化。

### 18.2 PlayMode

- 从新单局进入第一层地图。
- 选择小怪，完成商店、战斗、奖励并返回地图。
- 进入精英并完成三选一奖励。
- 进入强化节点后不触发商店或战斗。
- 事件、强化和恢复点选择后正确返回地图。
- Boss 失败后重试。
- 三层 Boss 胜利后进入通关结算。
- 生命归零后进入失败结算。
- 场景重载不会重复扣血、重复奖励或重复推进节点。
- 奖励队列按先进先出顺序领取，且单局结束时不泄漏牌池预留。

## 19. 分步落地顺序

### 19.1 阶段 4A：单层最薄闭环

先实现：

- `RunState` 和 `RunPhase`。
- 一张“强制首战 → 普通节点 A/B → Boss”的固定单层技术验证地图。
- 小怪和 Boss 两种战斗节点。
- 玩家生命和战斗扣血。
- 普通奖励、延迟商店资源和奖励暂存队列。
- Boss 失败重试；最终 Boss 胜利直接完成原型，不发功能奖励。
- 通关、失败结算。

目标是完成：

`地图 → 强制首战商店/战斗 → 奖励 → 普通节点 A/B 商店/战斗 → Boss 商店/战斗 → 结算`

### 19.2 阶段 4B：路线风险收益

增加：

- 精英节点。
- 事件节点。
- 强化节点。
- 恢复点。
- 两条有明显差异的路线。
- 精英高价值奖励，以及可供第一、二层 Boss 复用的奖励选择基础。

### 19.3 阶段 4C：三层完整单局

增加：

- 三层地图配置。
- 每层普通遭遇、精英和 Boss。
- 跨层奖励和难度递增。
- 正式启用第一、二层 Boss 奖励，使其在下一层商店和路线中产生实际价值。
- 最终通关统计。
- 10-20 分钟时长和初步数值调整。

## 20. 阶段 4 完成标准

- 玩家可以从空阵容开始完成三层单局。
- 地图至少包含小怪、精英、强化、事件、恢复点和 Boss。
- 每层至少存在一次有意义的路线二选一。
- 战斗失败产生清晰、可解释的生命压力。
- 精英奖励明显高于安全路线奖励。
- Boss 失败、重试、胜利和跨层流程均不会卡死。
- 卡牌奖励在备战区已满时不会丢失或阻塞单局。
- 一局控制在 10-20 分钟。
- 自动化测试全部通过。
- 完成至少一次人工通关和一次人工失败验收。

## 21. 已确认规则

本轮已经按以下方向收敛：

- 第一版使用固定地图、完整预览和固定 Boss。
- 地图通过 `IMapProvider` 提供，后续可以替换为基于种子的随机生成器。
- 独立商店节点从第一版移除。
- 每个地图节点和 Boss 重试都会推进 `RunTurn`；未打开商店的节点同样推进预算和升级降价进度。
- 初始生命采用 20，精英和 Boss 伤害修正采用 +2 / +4，数值全部配置化。
- 平局固定扣 1 点生命并按未胜利处理。
- 小怪胜利使用随机单项奖励，不打开选择界面。
- 精英失败后节点完成并允许继续前进。
- Boss 失败后通过新一轮商店重试，不设置固定重试次数。
- 卡牌奖励使用独立暂存区，在后续商店领取或跳过。
- 奖励暂存区使用无硬上限的先进先出队列；下一次商店结束前必须领取或跳过全部奖励，并正确管理有限牌池预留。
- 满生命时不能选择零收益恢复。
- 第三层 Boss 胜利后不发放功能奖励。
- 第一版只使用现有战斗关键词，不制作精英和 Boss 专属技能。
- 正式加入“锻造台”强化节点，每层最多一个。
- 锻造台第一批只开放“均衡锻造、Shield、Taunt”，暂缓 `Cleave`。
- 第一版强化候选由固定节点配置，不在进入时随机生成。
- 同一随从可在多个锻造台重复强化；关键词不可重复，+2/+2 可叠加，金色随从仍可强化。
- 恢复点采用“恢复 6 / 最大生命 +2 并恢复 2”的生命专属方案。
- 第一层以强制普通首战开局，首个敌人是一个低强度 1 级随从，首战后才开始分支。
- 每层实际访问 3-4 个节点（含 Boss），完整单局约 10-12 个节点。
- 节点尝试使用 `NodeAttemptId` 和分项提交标记保证跨场景结算幂等。
- 战斗结果区分正常胜负、双方同时倒下和回合上限平局。
