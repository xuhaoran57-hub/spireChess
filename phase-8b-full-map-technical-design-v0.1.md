## 1. 背景

当前三层地图已经完成显式商店、完整单局和 Boss 失败重试闭环，但仍属于阶段 4D 的紧凑版本：每层任意路线固定经过 5 个商店、5 场战斗和 1 个功能节点，普通精英位于 C2。遗珍和事件战斗接入后，这个节奏暴露出三个问题：

1. 第一层玩家只经历 2 个商店和 1 场战斗就可能面对精英，构筑与生命准备不足。
2. 每层只有 4 场非 Boss 固定战斗，难以同时容纳教学、机制预告、路线抉择和 Boss 前检验。
3. 遗珍按 `ShopTurn` 和未来战斗数持续产生收益，地图长度已经成为平衡规则的一部分，不能继续作为散落在校验器和测试中的常量。

阶段 8B 将地图扩展为正式的三层固定手工地图，并把路线长度、风险收益和未来随机生成边界变成显式领域规则。

## 2. 已冻结结论

以下结论属于阶段 8B 硬边界：

- 每层固定 6 个商店、6 场固定战斗，其中 5 场非 Boss，C6 为 Boss。
- 商店与固定战斗继续严格交替；功能节点不推进 `ShopTurn`。
- 常规精英只在 C4 或更晚出现，首版固定在 C4 的强攻路线。
- 每条完整路线恰好经过 1 个功能节点。
- C2 和 C5 各提供 2 个同奖励档、不同敌方机制的普通战斗选择。
- C4 提供强攻、奇遇、保守三条风险收益路线，之后重新汇合。
- 遇怪事件属于可选额外战斗，不占用固定 6 战的 `combatIndex`。
- 首版使用三层固定手工地图，不实现随机地图生成器。
- 保留 `IMapProvider` 边界；未来随机地图只能从审核过的拓扑模板生成。

这些规则优先于阶段 4D 文档中的 5×5 节奏。阶段 4D 文档继续记录当时的实现基线，不回写历史结论。

## 3. 目标与非目标

### 3.1 目标

1. 建立三层正式固定地图，每条路线的经济次数和固定战斗次数完全一致。
2. 将精英推迟到玩家已经历 4 个商店、3 场战斗之后。
3. 让路线选择分别承担永久战力、高波动收益和生命容错，而不是通过额外商店制造强弱。
4. 参数化地图节奏，消除领域校验、内容校验、UI 和测试中的 5×5 硬编码。
5. 对遗珍触发、事件生命价值、单局时长和遭遇曲线变化给出可验证预算。
6. 保持地图物化、重载、Boss 重试和事件战斗的确定性。
7. 让固定地图与未来随机地图共用同一份地图定义、校验器和路线预算报告。

### 3.2 非目标

- 不在阶段 8B 首版实现 `SeededGeneratedMapProvider`。
- 不生成任意有向无环图，不开放自由拼接节点。
- 不借地图扩展同时调整卡牌、遗珍核心效果或战斗规则。
- 不在没有 Unity 环境时完成正式地图 Prefab、连线动画、截图和人工体验验收。
- 不恢复 R17 Phase 6B 的旧 S1/S2 候选；地图和遗珍节奏变化后必须建立新候选。
- 不把事件遇怪改造成地图上的隐藏固定战斗。

## 4. 术语与计数口径

| 术语 | 定义 |
| --- | --- |
| 商店 | 地图上的 `Shop` 节点；进入成功后 `ShopTurn + 1` |
| 固定战斗 | 类型为 `Normal`、`Elite` 或 `Boss` 的地图节点 |
| 额外战斗 | 事件选项通过 `followupEncounterId` 发起的战斗 |
| 战斗序号 | 固定战斗的层内 `combatIndex`，连续为 1–6 |
| 战斗尝试 | 一次实际模拟；Boss 重试会增加尝试，但不增加固定节点 |
| 功能节点 | `Enhance`、`Event`、`Rest` 之一 |
| 路线 | 从一个起点到唯一 Boss 的完整节点序列 |
| 路线角色 | `Aggressive`、`Adventure`、`Conservative` |

统计和文案必须明确区分：

- “本层固定战斗 4/6”只统计地图战斗节点。
- “本局总战斗”统计所有实际战斗尝试，包括事件战斗和失败重试。
- “每经过 N 个商店阶段”只由 `ShopTurn` 驱动，功能节点、事件战斗和 Boss 重试均不计数。

## 5. 单层标准拓扑

### 5.1 拓扑图

```text
S0 → C1 → S1 ─┬→ C2A ─┐
               └→ C2B ─┴→ S2 → C3 → S3 ─┬→ C4E 精英 → U-E 强化 ─┐
                                           ├→ C4N 普通 → U-V 事件 ─┼→ S4 ─┬→ C5A ─┐
                                           └→ C4L 较弱 → U-R 恢复 ─┘      └→ C5B ─┴→ S5 → C6 Boss
```

其中：

- `S0` 是本层第 1 个商店，`S5` 是 Boss 前商店。
- `C2A/C2B` 和 `C5A/C5B` 是横向机制选择，不改变奖励档位。
- `C4E/C4N/C4L` 是唯一的纵向风险收益选择。
- `U-E/U-V/U-R` 分别是强化、事件、恢复，只能由对应 C4 分支进入。
- 三条 C4 分支在 `S4` 汇合，避免路线差异继续放大商店次数。

### 5.2 数量

每层固定配置包含 19 个可见节点：

| 节点组 | 可见数量 |
| --- | ---: |
| 商店 S0–S5 | 6 |
| C1 | 1 |
| C2A/C2B | 2 |
| C3 | 1 |
| C4E/C4N/C4L | 3 |
| U-E/U-V/U-R | 3 |
| C5A/C5B | 2 |
| C6 Boss | 1 |
| 合计 | 19 |

任意实际路线访问 13 个节点：

```text
6 商店 + 6 固定战斗 + 1 功能节点 = 13
```

C2、C4、C5 的组合共形成 `2 × 3 × 2 = 12` 条合法 Boss 路线。三层共配置 57 个可见节点，完整无额外战斗路线访问 39 个节点。

### 5.3 精英时机

进入 C4 精英之前，玩家已经完成：

- S0、S1、S2、S3，共 4 个商店；
- C1、C2、C3，共 3 场固定战斗。

精英之后仍保留功能节点、S4、C5、S5 和 Boss。玩家既有足够发育时间，也能在后续两个商店中消化精英奖励和强化结果。

## 6. 战斗档位与三层主题

### 6.1 层内职责

| 序号 | 职责 | 内容约束 |
| --- | --- | --- |
| C1 | 基线检查 | 低复杂度，验证当前阵容基本战力 |
| C2 | 机制预告 | 两个普通遭遇同奖励、相近威胁，不提供精英奖励 |
| C3 | 中段检查 | 组合 1–2 个本层核心机制 |
| C4 | 路线抉择 | 精英 / 标准普通 / 较弱普通，对应三种功能收益 |
| C5 | Boss 前检验 | 两个普通遭遇同奖励、相近威胁，分别预告 Boss 的不同压力 |
| C6 | Boss | 本层唯一 Boss；第一、二层产出冠冕级遗珍 |

增加的是一个“非 Boss 固定战斗档位”，不是简单复制旧遭遇。现有遭遇需要按新 C1–C6 曲线重排；每条实际路线不得在同层重复相同 `encounterId`。若现有配置不足以支持所有可见分支，应补充普通遭遇，而不是复用同一敌阵伪装成选择。

### 6.2 三层主题

| 楼层 | 教学与压力主题 | C5/Boss 前重点 |
| --- | --- | --- |
| 第一层 | 基础站位、嘲讽、护盾和低强度召唤 | 让玩家理解目标选择与空位管理 |
| 第二层 | 溅射、破阵、法术/成长联动 | 检查阵容宽度和站位抗性 |
| 第三层 | 亡语、召唤、残局铺场和混合机制 | 检查完整构筑上限，不再引入新基础规则 |

具体属性值不在本方案中冻结。实施时以当前 5.4.1 遭遇为基线重新标定，要求难度总体随 C1 → C6 上升；C2 和 C5 的横向选项只允许机制不同，不允许存在肉眼可见的数值优选项。

## 7. 路线预算

### 7.1 结构预算

每条路线必须满足完全相同的结构预算：

| 预算项 | 数量 |
| --- | ---: |
| 商店 | 6 |
| 固定战斗 | 6 |
| Boss | 1，且为 C6 |
| 功能节点 | 1 |
| 精英 | 0 或 1；若存在不得早于 C4 |
| 实际访问节点 | 13 |

结构预算由 `MapValidator` 对全部 12 条路径逐条验证，不能只验证一条样例路线。

### 7.2 C2 与 C5 横向选择

C2 和 C5 的两个选项遵守：

- 都是 `Normal`；
- 使用相同奖励档，首版优先直接引用相同 `rewardTableId`；
- 威胁区间相近，差异主要来自嘲讽、护盾、召唤、溅射等机制；
- 地图预览明确展示机制标签，不能依赖玩家记忆敌人名称；
- 不与强化、事件、恢复绑定；
- 不改变后续可达节点。

如果模拟显示同序号两个选项的平均生命损失相差超过 1，或胜率相差超过 8 个百分点，应优先调整遭遇，而不是提高弱侧奖励；否则横向选择会重新变成风险收益选择。

### 7.3 C4 三条路线

| 路线 | 战斗 | 功能节点 | 主要收益 | 主要代价 |
| --- | --- | --- | --- | --- |
| 强攻 `Aggressive` | C4E 精英 | 强化 | 精英奖励 + 可控永久战力 | 最高失败和生命损失风险 |
| 奇遇 `Adventure` | C4N 标准普通 | 事件 | 高方差资源、生命交易、遗珍或额外战斗 | 结果不稳定，可能零收益或额外受伤 |
| 保守 `Conservative` | C4L 较弱普通 | 恢复 | 生命与最大生命容错 | 最低永久战力上限 |

防止严格支配的约束：

1. C4L 的战斗奖励不得高于 C4N；保守路线不能同时更安全、恢复更多且获得相同永久收益。
2. C4E 必须提供显著高于普通路线的永久战力上限，才能补偿精英风险。
3. 事件始终保留合法退出，但有收益选项的生命代价和随机性必须计入路线预算。
4. 强化、事件和恢复的价值不通过额外商店补偿。
5. 路线只在 C4–U 段产生差异，S4 后重新回到同一节奏。

### 7.4 预算报告

结构合法不等于数值平衡。实施时生成每层 12 条路线的机器可读报告，至少包含：

```text
floor / pathId / c2Choice / routeRole / c5Choice
shopCount / fixedCombatCount / utilityCount / eliteCount
combatRewardTables / utilityType
winRate / averageHealthLoss / bossReachRate
expectedPermanentStats / expectedGoldEquivalent
eventCombatRate / curioAcquireRate
```

`MapValidator` 负责结构错误；离线模拟和报告负责内容预算。不得把主观“强/中/弱”写进校验器代替模拟结果。

## 8. 事件战斗

事件节点仍是 `Event`，其遇怪选项通过现有 `followupEncounterId` 进入战斗：

1. 进入事件时物化事件和选项。
2. 选择遇怪后，在同一个 `NodeAttemptState` 中物化后续遭遇。
3. 战斗使用独立、确定性的事件战斗随机流。
4. 胜负结算后返回事件节点的完成流程，再回到地图。
5. 事件战斗前已经存在的 FIFO 奖励继续保留，不能被覆盖。

计数规则：

- 不为事件战斗设置 `combatIndex`，也不占用 C1–C6。
- 不开启商店，不增加 `ShopTurn`，不推进升级降价。
- 计入本局实际战斗、胜负和耗时统计。
- 不计入“固定战斗 6/6”和地图节奏校验。
- 每层可能因此出现第 7 场战斗；三层固定路径为 18 战，事件最多再增加 3 个地图外战斗。
- Boss 失败重试同样只增加战斗尝试，不增加固定战斗节点。

事件遭遇的强度必须按发生在 C4 之后标定，不能继续沿用早期事件位置的敌方身材。

### 9.3 事件遗珍剩余窗口

事件位于 C4 后、S4 前。若在该位置获得奇物级遗珍：

| 获取楼层 | 剩余商店 | 剩余固定战斗 |
| --- | ---: | ---: |
| 第一层 | 14 | 14 |
| 第二层 | 8 | 8 |
| 第三层 | 2 | 2 |

第三层只有两个商店，E07 征募铜章无法完成 3 商店周期，延迟收益型奇物的价值明显低于即时效果。实施地图时必须完成下列二选一模拟决策，但本技术方案不直接修改数值：

- 将第三层奇物生命代价从 3 降为 2；或
- 第三层事件池/候选池偏向即时生效遗珍，避免生成本局无法触发的周期效果。

无论采用哪种方式，都要保持“至少保留 1 点生命”和候选原子提交规则。

## 10. 配置模型

### 10.1 地图规则档案

`run-maps.v0.1.json` 升级时增加可复用规则档案，固定地图和未来生成地图引用同一规则：

```json
{
  "ruleProfiles": [
    {
      "id": "full_map_6x6_v1",
      "shopCount": 6,
      "combatCount": 6,
      "bossCombatIndex": 6,
      "eliteMinCombatIndex": 4,
      "utilityCountPerPath": 1,
      "requireShopCombatAlternation": true
    }
  ],
  "maps": [
    {
      "id": "phase8b_floor1",
      "floor": 1,
      "ruleProfileId": "full_map_6x6_v1",
      "startNodeIds": ["f1_shop_start"],
      "nodes": []
    }
  ]
}
```

领域模型增加：

```text
MapRuleProfile
├── Id
├── ShopCount
├── CombatCount
├── BossCombatIndex
├── EliteMinCombatIndex
├── UtilityCountPerPath
└── RequireShopCombatAlternation

MapDefinition
├── Id / Floor
├── RuleProfile
├── SourceInfo
├── Nodes
└── StartNodeIds
```

`SourceInfo` 至少记录 `providerId`、`generationVersion`、`generationSeed` 和 `mapHash`。固定地图的生成种子可为空，但仍要有稳定哈希。

### 10.2 路线标签

C4 分支节点增加可选 `routeTag`：

- `Aggressive`
- `Adventure`
- `Conservative`

该字段只用于路线预览、遥测和预算分组，不参与实际奖励结算。奖励仍由节点类型和 `payloadId` 决定，避免 UI 标签成为规则输入。

### 10.3 遭遇配置

本阶段不把 `riskText` 当成机器可读难度。结构校验只验证：

- `Normal` 节点引用 `category = Normal` 的同层遭遇；
- `Elite` 节点引用 `category = Elite` 的同层遭遇；
- `Boss` 节点引用 `category = Boss` 的同层遭遇；
- C2A/C2B、C5A/C5B 使用相同奖励档；
- 所有战斗节点的 `combatIndex` 与路径位置一致。

威胁差异由离线模拟报告判定。未来如需要自动组装遭遇，再单独引入稳定的 `threatBand`，不在本阶段用敌方总身材替代真实战斗强度。

## 11. 地图提供者与随机生成扩展口

### 11.1 领域边界

```text
GameApp / Run composition root
              ↓ 注入
         IMapProvider
          ├── FixedMapProvider
          └── SeededGeneratedMapProvider（后续）
              ↓
         MapDefinition
              ↓
          RunSession
```

`RunSession` 只做两件事：

1. 新开局或进入下一层时用 `MapRequest(seed, floor)` 请求地图。
2. 将返回的 `MapDefinition` 写入 `RunState` 并按统一规则运行。

`RunSession` 不判断固定/随机模式，不读取生成模板，不根据 Provider 类型走分支。当前为兼容测试保留的“未注入时自动创建 `FixedMapProvider`”入口应逐步收敛到组合根，正式运行路径必须显式注入。

### 11.2 后续随机生成原则

未来 `SeededGeneratedMapProvider` 只允许：

- 从审核过的 6×6 拓扑模板中选择；
- 为模板槽位选择合法的遭遇和功能内容；
- 使用 `SeedDeriver` 从单局种子、楼层、Provider 版本派生独立地图种子；
- 生成稳定节点 ID、连线、位置和 `mapHash`；
- 输出后通过与固定地图相同的 `MapValidator` 和路线预算检查。

明确禁止：

- 任意 DAG；
- 先随机节点再尝试修补成合法路线；
- 让某条路线多一个商店或少一个固定战斗；
- 在重载、回到地图或 Boss 重试时重新生成；
- 因商店刷新、事件或战斗随机调用改变地图结果。

## 12. 物化、重载与确定性

1. 地图只在新开局和 `ContinueToNextFloor()` 成功时物化一次。
2. 物化后的完整 `MapDefinition` 是当前楼层的事实来源；场景重载直接复用，不再次调用 Provider。
3. 未来存档必须保存完整地图快照和 `mapHash`，不能只保存种子。否则内容热更新会使旧存档重抽地图。
4. 地图随机流与商店、事件、奖励、遗珍、普通战斗和事件战斗随机流相互独立。
5. 同一运行种子、楼层、Provider 版本和内容版本必须得到相同地图哈希。
6. Provider 或内容版本变化允许产生新地图，但不能改变已经物化的当前楼层。
7. Boss 重试继续复用同一节点尝试和战斗种子，不重新开商店、不增加 `ShopTurn`、不发放永久成长或战后奖励。

## 13. 校验规则

### 13.1 `RunContentValidator`

负责配置和引用完整性：

- 规则档案 ID 唯一、数值为正且 `bossCombatIndex <= combatCount`。
- 每层恰好有一个正式地图并引用存在的规则档案。
- 节点 ID 唯一，`payloadId` 指向正确内容集合。
- 战斗 `combatIndex` 范围由规则档案决定，不再硬编码 1–5。
- 节点类型与遭遇 `category`、`floor` 匹配。
- C2/C5 横向节点奖励档一致。
- C4 三个分支的 `routeTag` 唯一且完整。
- 非战斗节点 `combatIndex = 0`。

### 13.2 `MapValidator`

负责图和每条路径：

- 起点存在且为 `Shop`。
- 节点和边均存在，图无环。
- 所有节点从某个起点可达，且都能到达唯一 Boss；禁止孤儿节点和死路。
- Boss 无后继，类型唯一，`combatIndex = 6`。
- 枚举的每条 Boss 路线均为 13 个节点。
- 忽略功能节点后的显著序列严格为：

```text
Shop → C1 → Shop → C2 → Shop → C3 →
Shop → C4 → Shop → C5 → Shop → C6 Boss
```

- 每条路线恰好 6 商店、6 固定战斗、1 功能节点。
- 战斗序号严格为 1、2、3、4、5、6。
- 精英不得早于 `eliteMinCombatIndex`，每条路线最多一个精英。
- 路径数量设置安全上限；模板异常导致组合爆炸时校验失败，不无限枚举。

### 13.3 `RouteBudgetValidator / Report`

负责阶段 8B 专属结构和报告：

- 固定地图恰好有 12 条 Boss 路线。
- 三种 `routeTag` 均可达且在 S4 前汇合。
- 强攻路线包含一个精英和一个强化节点。
- 奇遇路线包含一个普通 C4 和一个事件节点。
- 保守路线包含一个普通 C4 和一个恢复节点。
- C2/C5 的两个选择都是普通战斗且奖励档一致。
- 输出第 7 节定义的预算表，供模拟和人工复核。

通用 `MapValidator` 不应硬编码“恰好 12 条路线”或具体功能组合；这些由 `full_map_6x6_v1` 的预算校验器承担。

## 14. 状态机与领域流程

地图扩展不新增 `RunPhase`。现有流程继续成立：

```text
MapSelection
  → Shop
  → Battle / EventChoice / EnhanceChoice / RestChoice
  → BattleResult / RewardChoice / RelicChoice
  → MapSelection / FloorComplete / RunWon / RunLost
```

需要保持的边界：

- `MapStep` 只在成功进入地图节点时增加；完整无额外战斗单局为 39。
- `ShopTurn` 只在成功进入商店时增加；完整单局为 18。
- 功能节点和事件战斗不会改变升级降价时钟。
- 第一、二层 Boss 冠冕选择完成后才能进入 `FloorComplete`。
- 第三层 Boss 仍直接进入 `RunWon`，不生成本局无法消费的冠冕遗珍。
- 节点奖励、事件奖励和遗珍自动卡牌继续共用 FIFO 队列，来源尝试不能相互覆盖。
- 事件战斗结算后完成原事件节点，不将事件节点永久改写成 `Normal`。
