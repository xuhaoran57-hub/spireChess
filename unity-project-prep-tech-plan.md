# Unity 项目准备阶段技术方案

版本：0.1
关联文档：`game-design.md`、`development-plan.md`
当前阶段：阶段 0，项目准备
目标平台：PC、Android、iOS，预留微信小游戏技术验证路线
表现形式：2D 卡牌式
数据配置：JSON
存档：从第一版开始支持
核心数据结构：暂不定稿，待讨论后更新

## 1. 阶段目标

本阶段的目标不是开始堆玩法内容，而是把 Unity 项目的技术底座搭好，让后续战斗、商店、三连、爬塔地图、存档和多平台打包都能在同一套架构上推进。

阶段 0 完成后，应达到以下状态：

- Unity 项目可以在 PC 编辑器中稳定运行。
- 项目目录结构清晰，后续代码、配置、UI、美术资源有固定放置位置。
- JSON 配置加载链路跑通。
- 存档读写链路跑通。
- 2D 卡牌式 UI 的基础技术方案明确。
- PC 和移动端输入方案有统一抽象。
- 微信小游戏适配被列为技术探针，不阻塞 MVP 主线。
- 核心数据结构章节保留空位，待玩法规则讨论完后补全。

## 2. 推荐 Unity 技术选型

### 2.1 Unity 版本

建议使用 Unity LTS 版本，优先考虑：

- Unity 2022 LTS：生态稳定，插件兼容性较好。
- Unity 6 LTS：如果后续要长期维护，可以考虑，但需要先确认微信小游戏转换链路和移动端插件兼容性。

MVP 阶段建议优先选 Unity 2022 LTS，理由是稳定、资料多、风险低。

### 2.2 渲染方案

推荐使用 2D URP 或内置渲染管线。

MVP 建议：

- 如果团队熟悉 URP：使用 URP 2D Renderer。
- 如果追求最少变量：使用内置渲染管线。

由于当前项目是 2D 卡牌式表现，不需要 HDRP，不需要复杂 3D 光照，不建议引入重型渲染方案。

### 2.3 UI 方案

推荐使用 UGUI。

理由：

- 适合卡牌、商店、地图、弹窗、三选一奖励等界面。
- 触屏和鼠标交互都成熟。
- 拖拽、点击、长按、Tooltip 等功能实现直接。
- 与移动端和 PC 都兼容。

MVP 阶段不建议使用 UI Toolkit 作为主 UI，因为项目需要大量运行时拖拽、卡牌动态生成和移动端适配，UGUI 更稳妥。

### 2.4 语言与代码风格

语言：C#。

建议约束：

- 玩法逻辑尽量写成普通 C# 类，减少对 MonoBehaviour 的依赖。
- MonoBehaviour 主要负责生命周期、场景挂载、UI 绑定和表现层调用。
- 战斗结算、商店刷新、三连判断、奖励生成等逻辑应尽量可单元测试。
- 数据对象和运行时对象分开，避免直接修改配置数据。

## 3. 平台目标与发布策略

### 3.1 PC

PC 是主开发和调试平台。

用途：

- 快速验证战斗。
- 快速调试商店和经济。
- 快速读取本地 JSON。
- 快速导出测试包。

优先保证 Windows PC 版本稳定。macOS 和 Linux 可以后续再确认。

### 3.2 移动端

移动端从早期就要考虑，但不要求第一周完成正式适配。

需要提前决定：

- 横屏还是竖屏。
- 卡牌尺寸和可读性。
- 拖拽站位是否适合手指操作。
- 商店按钮是否足够大。
- Tooltip 在移动端如何触发。

建议 MVP 使用横屏，理由是：

- 5 个战斗位更容易展示。
- 商店、备战区、敌我双方阵容更容易排布。
- 更接近自走棋和卡牌战斗的观察习惯。

### 3.3 微信小游戏

微信小游戏不作为阶段 0 的主交付目标，但需要做技术探针。

建议策略：

- 主工程先以 PC 和移动端为基础设计。
- 所有平台相关能力封装接口，例如存档、文件读取、音频、输入。
- 在阶段 0 或阶段 1 末尾创建一个极小的微信小游戏打包验证分支或场景。
- 不要等完整 MVP 做完后才验证微信小游戏，否则返工风险高。

微信小游戏探针最低要求：

1. 启动一个空场景。
2. 读取一份 JSON 配置。
3. 显示 5 张卡牌。
4. 支持鼠标/触摸拖拽换位。
5. 播放一个音效。
6. 写入和读取一份简单存档。
7. 在微信开发者工具中运行。
8. 在真机中运行。
9. 记录首包大小、启动时间、内存占用和帧率。

## 6. 程序架构建议

建议采用轻量分层架构，不引入过重框架。

### 6.1 分层方式

```text
表现层 UI/View
  调用
应用层 Controller/Presenter
  调用
玩法领域层 Domain Logic
  读取
配置层 Config Data
  保存/读取
存档层 Save Data
```

各层职责：

- 表现层：显示卡牌、按钮、动画、拖拽反馈。
- 应用层：连接 UI 和玩法逻辑，处理玩家操作。
- 玩法领域层：战斗、商店、三连、奖励、地图等纯规则。
- 配置层：加载 JSON 配置，提供只读数据。
- 存档层：保存玩家当前局状态。

### 6.2 服务划分

建议阶段 0 先规划以下服务类：

- `GameApp`：全局启动入口。
- `ConfigService`：配置加载和查询。
- `SaveService`：存档读写。
- `SceneFlowService`：场景切换。
- `InputService`：鼠标和触摸输入抽象。
- `AudioService`：音效和音乐播放。
- `RandomService`：统一随机数。
- `LogService`：调试日志和战斗日志。

MVP 初期可以用简单单例或启动场景挂载对象，不建议一开始引入复杂依赖注入框架。

### 6.3 玩法模块划分

建议代码按模块拆分：

- `Battle`：自动战斗、目标选择、技能触发、战斗日志。
- `Shop`：刷新、购买、出售、冻结、商店池。
- `Economy`：金币、费用、奖励结算。
- `Tavern`：酒馆等级、升级费用、随从等级池。
- `Collection`：玩家战斗区、备战区、法术区。
- `Triple`：三连检测、金色随从、发现牌。
- `Rewards`：奖励池、三选一、Boss 奖励。
- `Map`：地图生成、节点选择、路线推进。
- `Run`：单局状态管理。
- `Save`：存档序列化和反序列化。

## 7. JSON 配置方案

### 7.1 配置文件来源

MVP 阶段建议把 JSON 放在 `Assets/Resources/Configs` 或 `StreamingAssets` 中。

推荐阶段策略：

- 阶段 0-2：使用 `Resources`，加载简单，适合快速原型。
- 阶段 3 以后：如果配置越来越多，再评估迁移到 Addressables 或 StreamingAssets。
- 微信小游戏探针：单独验证目标平台对文件读取方式的支持。

### 7.2 JSON 解析方案

Unity 自带 `JsonUtility` 对复杂 JSON 支持有限，尤其是字典、多态、嵌套数组处理不方便。

建议：

- 简单 MVP 可以先用 `JsonUtility`。
- 如果后续字段复杂，使用 Newtonsoft Json for Unity。

阶段 0 建议先做一层 `IJsonSerializer` 封装，避免后续替换解析库影响业务代码。

示例职责：

```text
IJsonSerializer
  FromJson<T>(string json)
  ToJson<T>(T value)
```

### 7.3 配置加载流程

启动时加载：

1. Boot 场景启动。
2. `ConfigService` 读取所有 JSON 文本。
3. 反序列化为配置对象。
4. 校验配置合法性。
5. 建立按 ID 查询的索引。
6. 进入主菜单或测试场景。

### 7.4 配置校验

阶段 0 应先定义校验机制，不必一次写完所有规则。

建议校验方向：

- ID 不为空。
- ID 不重复。
- 引用的 ID 存在。
- 数值不为负。
- 酒馆等级范围合法。
- 奖励池不为空。
- 文本字段不为空。

配置错误应在启动时明确打印，而不是进入游戏后才报错。

## 9. 存档技术方案

### 9.1 存档目标

从第一版开始支持存档，但阶段 0 只需要做最小可用链路。

最小存档内容：

- 存档版本号。
- 创建时间和更新时间。
- 一个简单测试字段，例如金币数量。
- 一组测试随从 ID。

阶段 4 以后再扩展为完整单局存档。

### 9.2 存档位置

PC 和移动端使用：

```text
Application.persistentDataPath
```

建议文件：

```text
save_slot_0.json
settings.json
```

微信小游戏需要单独适配微信文件系统或本地存储，因此必须通过接口封装。

### 9.3 存档接口

建议定义：

```text
ISaveStorage
  bool Exists(string key)
  string ReadText(string key)
  void WriteText(string key, string content)
  void Delete(string key)
```

实现类：

- `FileSaveStorage`：PC、Android、iOS 使用。
- `WechatSaveStorage`：微信小游戏探针阶段再实现或接入。
- `MemorySaveStorage`：测试使用。

### 9.4 存档版本

存档必须带版本号。

原因：

- MVP 阶段数据结构会频繁变化。
- 没有版本号，旧存档很容易导致难以定位的问题。

初版策略：

- 版本不兼容时允许清空旧存档。
- 等系统稳定后再做迁移逻辑。

## 11. 2D 卡牌式表现方案

### 11.1 卡牌预制体

建议卡牌 Prefab 拆成：

- 背景框。
- 头像区域。
- 名称文本。
- 攻击力文本。
- 生命值文本。
- 等级标记。
- 关键词图标区域。
- 技能描述文本。
- 金色边框状态。

### 11.2 战斗区域

建议基础布局：

- 上半屏：敌方 5 个战斗位。
- 中间：战斗日志、攻击动画区域或分隔区。
- 下半屏：玩家 5 个战斗位。
- 底部：备战区和可用法术。

如果移动端横屏：

- 需要保证卡牌文字不要过小。
- 技能全文可以通过详情弹窗展示。
- 卡面只显示核心信息。

### 11.3 商店区域

建议基础布局：

- 中央：4 个随从商品位。
- 右侧或底部：1-2 个法术商品位。
- 固定区域：金币、酒馆等级、刷新、升级、冻结。
- 下方：玩家战斗区和备战区。

### 11.4 地图区域

MVP 地图可以先用简单节点图：

- 节点图标。
- 路线连线。
- 当前节点高亮。
- 可选下一节点高亮。

正式表现可以后续再美化。

## 14. 包和插件建议

### 14.1 阶段 0 建议少装插件

MVP 阶段应尽量减少第三方依赖。

可考虑：

- TextMeshPro：Unity 常规文本方案。
- Newtonsoft Json for Unity：如果 JsonUtility 不够用再加入。
- DOTween：如果需要快速做 UI 动画，可在阶段 7 前后加入。

暂缓：

- 大型 UI 框架。
- 复杂状态机框架。
- 重型依赖注入框架。
- 商业化 SDK。
- 热更新框架。

### 14.2 文本方案

使用 TextMeshPro。

注意：

- `Builds/`
- `Logs/`
- `UserSettings/`

建议提交：

- `Assets/`
- `Packages/`
- `ProjectSettings/`

分支建议：

- `main`：稳定版本。
- `develop`：日常开发。
- `prototype/battle`：战斗原型。
- `prototype/wechat-minigame`：微信小游戏探针。

个人开发可以简化为 `main` + 功能分支。

## 19. 当前建议结论

基于当前项目目标，推荐技术路线为：

- Unity 2022 LTS。
- C#。
- UGUI。
- TextMeshPro。
- 2D 卡牌式表现。
- PC 作为主开发平台。
- Android/iOS 从 UI 和输入阶段开始适配。
- JSON 配置先走简单加载，后续按复杂度决定是否引入 Newtonsoft。
- 存档从第一版开始封装接口，PC 和移动端先用 `Application.persistentDataPath`。
- 微信小游戏作为早期技术探针，不建议在未验证前承诺为 MVP 必达发布平台。

下一步最应该讨论的是核心数据结构，尤其是随从、法术、敌人、奖励和存档的字段。字段定清楚后，就可以正式进入 Unity 项目初始化和阶段 1 战斗原型。
