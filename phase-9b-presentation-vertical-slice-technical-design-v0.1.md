## 1. 背景与结论

阶段 7 至 9A 已经完成共享卡牌、商店、战斗、地图、主菜单和存档恢复的工程结构。现有 Prefab、ViewModel、View、Controller、结构化战斗事件和双分辨率布局可以作为正式表现的承载层，但当前视觉仍以纯色、文字和占位图形为主：

- 67 个随从配置全部使用 `placeholder_*` 插画 ID，当前仅不熄炉王映射到工程样板 Sprite；67 个 `audioId` 均为空；
- 16 张法术全部使用占位插画 ID，16 个 `audioId` 均为空；
- 15 件遗珍已有语义图标 ID，但项目中尚无对应 Sprite；
- `sc/Assets/Art` 已有 28 项清理生产许可的活动候选，其中部分仅有离线 master、部分已作 G2 技术预接入；材质、动画、音效和 BGM 仍无已清生产许可的正式资产，且任何候选在 G2 验收前都不是 Runtime Ready；
- `PF_Card`、`PF_ShopScreen`、`PF_BattleScreen`、`PF_RunScreen` 和 `PF_MainMenuScreen` 已具备信息与交互契约，但不代表最终美术品质。

9B 不直接生产全部内容。先以第一层 10–15 分钟体验为纵向切片，冻结美术方向、资产规格、运行时接线、通用反馈和音频管线，再决定是否进入全量资产生产。自动平衡 S0/S1 可以与表现制作并行；依赖体验判断的 S2 人工单局应在 9B 第一版完成后执行。

## 2. 目标、成功标准与非目标

### 2.1 目标

1. 将 `主菜单 → 地图 → 商店 → 普通战斗 → 奖励 → 返回地图` 做成第一段接近目标品质的完整流程。
2. 用 12 张代表随从覆盖三大流派、普通/金色、低/高等级、护盾、战吼、亡语、召唤、刷新、施法、成长和阻塞选择。
3. 建立卡牌插画、UI Sprite、材质、动画/VFX、音效和 BGM 的命名、导入、引用、验收与来源记录规范。
4. 让正式资产只改变表现，不改变战斗、经济、随机流、配置身份或存档语义。
5. 建立可以继续批量生产 64 个非 Token 随从、3 个 Token、16 张法术和 15 件遗珍的流水线。

### 2.2 成功标准

- 指定流程中不存在纯色方块或缺失引用；未制作专属插画的卡牌使用明确的种族剪影回退图，不显示空白或开发占位图。
- 12 张样板随从和 3 个关联 Token 在 Full/Compact、普通/金色、商店/战斗/选择层中身份一致。
- 购买、出售、刷新、冻结、升级、上场、施法、三连、发现、攻击、受伤、护盾、属性变化、死亡、召唤和胜负均有可识别反馈。
- 主菜单、地图/商店、普通战斗至少各有一套可循环 BGM；Master/Music/SFX/UI 四条音量通道可独立调节。
- 1920×1080、1920×1200 下无核心遮挡、裁切、文字不可读或特效越界。
- 表现层不消费领域 RNG，不触发第二次模拟，不改变现有确定性哈希；跳过表现仍提交同一次战斗结果。
- 在开发机和目标测试机上完成性能基线记录；目标设备未冻结前不以单一机器帧率替代性能结论。
- 至少邀请 5 名未参与实现的人完成指定流程；不经口头解释也能辨认购买、上场、路线选择、攻击目标、护盾破裂和战斗结果。

### 2.3 非目标

- 一次性完成全部 64 个非 Token 随从、3 个 Token、16 张法术和 15 件遗珍的最终美术。
- 为每张卡建立独立动画、粒子系统、配音或专属音效。
- 修改卡牌规则、地图结构、经济、数值、随机流或存档 schema。
- 重做阶段 7/8 已冻结的信息架构和输入链路。
- 微信小游戏适配、正式发布包、云存档、局外成长或完整新手引导。
- 在风格样板尚未通过前批量生产同类资产。

## 3. 纵向切片范围

### 3.1 玩家流程

正式验收使用以下链路，不以单独 Preview Scene 代替：

1. 从 MainMenu 新建单局。
2. 在第一层地图选择起始商店节点。
3. 完成至少一次刷新、购买、上场、目标型法术或战吼、冻结/解冻和升级反馈。
4. 进入第一场普通战斗，完整播放攻击、伤害、护盾、死亡或召唤中的至少五类事件。
5. 处理战后奖励并返回地图。
6. 保存返回 MainMenu 后继续游戏，确认表现资源和状态可以重建。

除正式链路外，保留 Card/Shop/Battle/Run Preview Scene 作为风格评审和截图夹具。高等级样板卡不要求通过第一层自然牌池全部获得，使用既有测试/预览夹具验证，不为展示修改正式牌池。

## 4. 美术方向与评审门

### 4.1 共同方向

9B 的画面目标不是写实堆料，而是“暗色奇幻棋盘上的高对比卡牌收藏”。共同约束：

- 背景负责氛围，卡牌、按钮、节点和战斗目标负责信息；背景不得与可交互轮廓争夺亮度。
- 卡牌继续使用 2:3 外形、Full 240×360 和 Compact 160×240 的阶段 7 契约，不改变信息区域几何。
- T1–T5 灰、绿、蓝、紫、橙只作为等级识别强调，不大面积覆盖种族色和插画。
- 普通卡使用共享银黑结构，金色卡只替换公共框架和材质，不遮蔽种族皮肤、等级、名称或状态徽章。
- 所有装饰图形在 Compact 模式仍需可辨认；无法在 160×240 下成立的细节不进入公共 UI 规范。

### 4.2 三流派视觉关键词

| 流派 | 主色与辅色 | 形状语言 | 材质与动效 | 禁止项 |
| --- | --- | --- | --- | --- |
| 铸魂 | 炭黑、熔橙、暗红、少量冷钢 | 厚重几何、铆钉、环形炉纹、盾面切角 | 金属热变、火星、低频沉重脉冲 | 大面积亮红导致数值/错误色失去区分；细碎机械噪点 |
| 荒灵 | 苔绿、树皮褐、月青、暖黄 | 枝叶、蹄角、巢穴、流动曲线 | 花粉、叶片、呼吸式生长、轻快召唤轨迹 | 全屏高饱和绿色；Token 与本体轮廓过于接近 |
| 星契 | 深靛、银白、星青、折光紫 | 星盘、轨道、棱镜、书页和细圆弧 | 折射、轨迹线、计数环、清脆闪烁 | 过量粒子遮挡进度；与护盾蓝色完全同色 |

以上为 Style Tile 的约束，不等于最终插画风格。实施前必须用同一张卡、同一商店局部和同一地图节点制作至少 2 套风格样板，评审并冻结一种方向。

### 4.3 风格评审交付物

每套 Style Tile 至少包含：

- 铸魂盾侍普通 Full 卡；
- 天穹契约者普通/金色 Full 卡；
- 万蹄奔潮普通 Full 卡；
- 一个商店商品位、一个主要按钮、一个确认弹窗；
- 商店、普通战斗、事件三个地图节点；
- 攻击、生命、金币、护盾、亡语、战吼六个图标；
- 色板、字体层级、9-slice 边框、金色材质和背景明暗示例。

评审只回答四个问题：信息是否清楚、三个流派是否可区分、金色是否高级且可读、是否能以可接受成本批量生产。未通过评审不得开始其余 9 张样板插画。

## 5. 技术方案

### 5.1 保持领域边界

1. `BattleSimulator` 和各 Session 仍是唯一规则真源。
2. 表现系统只消费 `CardViewModel`、`ShopEventType`、`BattlePlaybackEventKind`、Run Screen State 和操作结果。
3. 表现随机只允许用于粒子角度、音高和无规则意义的装饰变化，使用独立的非领域随机源；不得读取或推进四条持久随机流。
4. 跳过、2× 速度、场景重载和存档恢复不得产生第二次领域操作。
5. 非播放平衡批次不加载或采集表现事件，保持阶段 6 成本与确定性门禁。

### 5.2 卡牌插画接线

当前 `CardViewModel` 没有插画身份，`CardView` 仍用种族纯色代替 Artwork。9B 采用以下最小扩展：

1. `CardViewModel` 增加只读语义字段 `ArtId`；随从/法术 Factory 直接从配置映射，不传 Sprite 或 Unity 对象。
2. 建立 `PresentationSpriteCatalog : ScriptableObject`，以稳定字符串 ID 映射 Sprite；禁止每次 Render 调用 `Resources.Load`。
3. `PF_Card` 序列化引用同一个 Catalog；`CardView` 按 `ArtId` 解析，未命中时按种族/法术类型使用明确的回退图并记录一次诊断。
4. 9B 保留现有 `placeholder_*` 配置 ID，不仅为改名修改配置哈希或让现有存档失效；Catalog 可以把该稳定 ID 指向正式样板 Sprite。
5. 正式样板卡的自动化门禁要求精确 ID 命中，不能用种族回退悄悄通过。

遗珍和地图图标使用同一个 Sprite Catalog 的语义 ID；音频使用独立 Catalog，不把 AudioClip 放入 CardViewModel。

### 5.3 主题与 UI 皮肤

建立最小 `PresentationTheme : ScriptableObject`，保存公共颜色、9-slice Sprite、普通/金色材质、种族皮肤、状态图标和屏幕背景引用。原则：

- 只集中真正跨屏复用的资源，不把每个 RectTransform 和文案配置抽象成主题系统。
- `PF_Card`、地图节点和通用按钮优先使用 9-slice Sprite，避免为不同尺寸复制位图。
- 现有 `CardTierPalette` 的等级语义保留，但颜色值迁移到主题资源；测试继续验证 T1–T5 可区分。
- View 不根据卡牌 ID写分支；卡牌专属差异来自 ArtId，种族差异来自主题，状态差异来自现有 ViewModel。

### 5.5 插画与 Sprite 规格

- 样板插画母版以 2048×2048、sRGB 制作，主体必须在中央 70% 安全区内，以适应 Full/Compact Artwork Mask 裁切。
- Unity 运行时导出为 1024×1024 PNG；Sprite、Mip Maps 关闭、Alpha Is Transparency 按实际资源设置，初期 Max Size 1024。
- UI 边框、面板和按钮优先使用透明 PNG + Border 的 9-slice；不得把文字烘焙进背景图。
- 金色版本默认不复制插画；普通与金色共享 Sprite，通过框架、材质、角饰和流光区分。
- 图标以 256×256 母版、128×128 运行时导出；小尺寸下使用粗轮廓，不依赖内部文字。
- 导入压缩在目标平台确定后统一冻结；9B 先记录无压缩/默认压缩的质量与内存对比，不逐文件随意设置。

### 5.6 动画与 VFX

现有 `BattlePlaybackEventKind` 已覆盖 CombatStarted、RoundStarted、AttackStarted、DamageApplied、ShieldGained、ShieldLost、StatsChanged、UnitDied、UnitSummoned、CombatEnded。9B 不新增战斗语义，只建立表现映射：

| 事件 | 第一版反馈 | 必须保持的规则 |
| --- | --- | --- |
| AttackStarted | 攻击者短距离突进、目标轮廓、轻微屏幕冲击 | 不改变槽位和目标；来源/目标使用稳定实例 ID |
| DamageApplied | 伤害浮字、目标闪白/震动 | `wasBlocked` 时不得播放生命损失反馈 |
| ShieldGained | 盾牌显现与蓝色脉冲 | 不遮挡已有状态徽章 |
| ShieldLost | 裂纹、碎片、盾牌消失 | 与 DamageApplied 顺序一致 |
| StatsChanged | `+X/+Y` 或负值浮字 | 复用实际差量，不从前后卡面重算规则 |
| UnitDied | 压暗、破碎/消散、清空槽位 | 先后顺序支持连续亡语 |
| UnitSummoned | 缩放入场、种族粒子、短音效 | 嵌套召唤不覆盖前一事件 |
| CombatEnded | 胜负遮罩、短结算音、返回按钮解锁 | 只提交一次已有结果 |

商店反馈以 `ShopEventType` 和操作结果映射刷新、购买、出售、上场、施法、三连、发现和升级。三连与发现可有较强演出，其余高频操作控制在 0.15–0.5 秒并允许连续输入，不让装饰动画阻塞领域操作。

通用 VFX 优先于单卡专属 VFX。9B 只为三个流派建立种族色粒子变体，不为 12 张卡各做独立特效。

### 5.7 音频架构

建立常驻音频服务与两个数据资源：

- `AudioMixer`：Master、Music、SFX、UI 四个 Group；音量设置独立于单局存档，使用本机设置存储。
- `PresentationAudioCatalog`：语义 Cue ID 对应 1–N 个 AudioClip、音量、音高范围、并发上限和冷却时间。
- `MusicDirector`：根据 MainMenu、Run/Shop、Battle 三个上下文淡入淡出；场景重载不叠加播放器。

9B 不填写每张卡的 `audioId`。购买、刷新、护盾、死亡等使用事件级通用 Cue；只有未来确需专属语音/音效的卡牌才使用配置中的 `audioId`，避免为 64 张卡制造无必要的音频债务。

音频导入基线：

- 短 SFX 保留 WAV 源，Unity 内按长度选择 Decompress On Load 或 ADPCM；高频短音不得使用高延迟 Streaming。
- BGM 运行时使用 Ogg Vorbis、Streaming，并验证无爆音循环点。
- 同一高频 Cue 提供 2–4 个轻量变体，音高/音量随机不得影响领域状态。
- 攻击、死亡、召唤和 UI 点击必须限制并发，避免嵌套亡语形成音量峰值。

### 5.8 来源、许可与生成式资产

所有外部或生成式资产必须在资产盘点表记录：作者/工具、模型或素材库、生成日期、许可证/商用范围、原始提示或来源链接、人工修改人和导出版本。无法证明来源或商用权利的资源只能作为概念参考，不得标为 Runtime Ready。

- 音频 Cue 的并发、冷却、Clip 引用和 Mixer Group 合法。

### 7.2 PlayMode

- 正式 MainMenu → Run → Shop → Battle → Run 流程使用新主题且只有一个 Canvas/EventSystem/音频服务。
- 刷新、购买、上场、施法、三连、发现和升级各触发一次正确表现；Render 不重复播放。
- 十类战斗结构化事件均能完成或安全跳过；2× 只缩放表现时间。
- 嵌套亡语与连续召唤不丢事件、不覆盖槽位、不形成无限音频叠加。
- 保存返回并继续后，卡牌插画、地图皮肤、BGM 上下文和音量设置正确恢复。
- Headless/平衡批次不依赖 Sprite、AudioClip 或 Animator，不改变现有确定性结果。

### 7.3 性能与资源预算

- 首次进入各正式场景记录加载耗时、总纹理内存、音频内存和首帧尖峰。
- CardView Render 不做磁盘/Resources 动态加载，不在每帧创建材质实例。
- 高频浮字、粒子和 AudioSource 使用有限复用池；战斗结束后无存活的临时实例。
- 同屏 10 张 Compact 卡、嵌套召唤和群体属性变化作为压力场景。
- 目标平台冻结后再写硬性帧时间和内存门槛；9B 先建立可重复基线并禁止明显回退。

## 9. 风险与控制

| 风险 | 控制 |
| --- | --- |
| 未定风格就批量画 64 张卡 | G1 前只制作 Style Tile；G1 后先完成 12 张样板 |
| 样板流程随机出现无专属插画卡 | 提供四种种族剪影回退；专项预览精确展示样板卡，不修改正式牌池 |
| 为金色卡重复全部插画导致成本翻倍 | 默认共享插画，用框架/材质区分；独立金色插画逐张立项 |
| 表现层重新计算规则或消费 RNG | 只消费 ViewModel/结构化事件；表现随机使用独立源；确定性回归 |
| 大量图片导致仓库和内存膨胀 | 源文件与运行时导出分离；先确定 LFS/外部库；统一 Max Size 和压缩 |
| 嵌套亡语产生粒子/音效风暴 | 复用池、并发上限、冷却和同类 Cue 合并策略 |
| UI 换肤破坏阶段 7 几何与点击 | 不改信息架构；Prefab 引用/几何/EventSystem 回归继续作为门禁 |
| 生成式资产版权或风格不一致 | 来源台账、商用许可、固定风格参考和人工修图评审 |
| 美术制作阻塞自动平衡 | S0/S1 与表现生产并行；S2 等 9B 第一版后执行 |
